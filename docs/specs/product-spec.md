## 0. エグゼクティブサマリー（結論先）

**「資産・口座・医療や葬儀の希望・家族へのメッセージを、誰のサーバにも送らず端末内だけで整理し、PDFとして遺せる、完全オフライン・買い切りの終活ノート」** を作る。

- **狙う隙間（確度：高）**：既存の終活アプリは大半がクラウド＋アカウント前提で、「サービス終了でデータ消失」「無料に見えて途中課金」「機微情報を事業者サーバに預ける不安」が繰り返し指摘されている。ここに **非送信・買い切り・PDF書き出し（＝終了しても紙で残る）** で対抗する。
- **支払い実績（確度：高）**：紙のエンディングノートは1,000〜3,000円で販売され、人は既にこの成果物に課金している。買い切り単価の妥当な根拠になる。
- **最大の前提（確度：中）**：クラウドの利便（自動同期・遠隔家族共有）を捨てる代償を、**PDF書き出し＋ユーザー自身のiCloud保存＋暗号化バックアップ** で補えるかがMVPの成否を分ける。
- **正直なリスク（確度：高）**：カテゴリは無料アプリで混雑し、大手系（MUFG等）も参入済み。発見性（ASO）は容易でない。収益目標¥100k/月の達成確度は中〜低（後述の試算参照）。
- **多言語の方針（確度：中）**：終活は制度・文化依存が強いため、単なるUI翻訳ではなく「**項目セットをロケールごとに出し分ける**」二層設計を採る。MVPは ja＋en の2言語で仕組みを確立し、テスト後にASC全言語へ拡大する。カスタム項目で各地域の不足を利用者自身が補える。

## 1. 背景と課題

紙のエンディングノートは「書き始めの心理的ハードル」「修正のしづらさ」「保管場所の問題」がある。一方、アプリ版は手軽だが、現状の主要アプリには以下の構造的弱点がある。

- **サービス終了リスク**：運営事業者の都合で予告なく終了し、保存データが失われうる（終了事例が実在）。死後に家族が初めてアクセスしようとして終了を知る最悪ケースがある。
- **課金導線の不透明さ**：無料で書き始めても、要所で「ここから有料」が出るアプリがある。
- **プライバシー不安**：資産・口座・ID/パスワード等、最も機微な情報を小規模事業者や金融機関のクラウドに預けることへの抵抗。

→ 本アプリは「**事業者のサーバに依存しない＝終了しても端末とPDFに残る**」を中核価値に据え、上記3点を同時に解消する。

## 4. プロダクト原則

1. **非送信を絶対遵守**：MVPはネットワーク権限を要求しない（技術的に送れない構成）。
2. **買い切り・広告ゼロ・データ収集ゼロ**。
3. **「終了に強い」**：データはユーザーの所有物。いつでもPDF／再取り込み可能なアーカイブで持ち出せる。
4. **やさしいUX**：Dynamic Type対応、平易な言葉、大きいタップ領域、専門用語回避。表示言語に追従。
5. **法的に踏み込まない**：情報の整理・記録のみ。法的・税務的結論は出さず、専門家相談を促す（第11章）。
6. **ロケール適応**：UI翻訳だけでなく、項目・免責・PDF体裁を地域の制度/文化に合わせて出し分ける（第6.7）。

## 6. 機能要件

### 6.1 項目設計（カテゴリ taxonomy）

**競合項目ベンチマーク（机上、確度：中）**：わが家ノート（16項目）、SouSou、つなぐノート（デジタル遺品特化）、市販紙ノート（看取り医監修）の公開情報を比較し、過不足を補正した。※これは公開情報ベースの机上比較であり、実機の使用感（入力負荷・UX）の最終評価はfreetokyo側での実機確認を推奨。

ベンチマークからの主な補強点：
- **「定期支払い・口座引き落とし」を独立項目化**（残された家族が継続課金を止められるように。わが家ノートも独立項目化）。
- **「カード・電子マネー・ポイント」「年金」を追加**（資産の取りこぼし防止）。
- **デジタル遺品を強化**（端末ロック解除方針、SNS/写真の扱い、暗証番号の“ありか”）。つなぐノートが特化する領域。
- **「自分史／略歴」を任意で追加**（SouSou・紙ノートが重視。義務化はしない）。
- **持ち歩ける緊急医療情報**（かかりつけ医・常用薬・アレルギー）をPDFの1枚として出力可能に。
- **意図的に外す**：家族共有・見守り・没後自動通知（オフライン方針のため。第15章Phase 4で再検討）。

### 6.2 機微情報の取り扱い方針（重要・差別化、確度：中）

口座番号・パスワード等の生データ保存は漏洩時リスクが高い。本アプリは二段設計を採る。

- **既定＝「ありかモード」**：秘密そのものではなく「どこにあるか（例：通帳は自宅金庫、パスワードは○○に記載）」を記録。安全側のデフォルト。
- **任意＝「秘匿モード」**：どうしても本体を記したい場合のみ、生体認証の再要求＋ **CryptoKitによる端末内暗号化** で保存。入力時に明確なリスク警告を表示。
- 秘匿項目はPDFやアーカイブ書き出し時に「含める／伏せる」を選択可能（既定は伏せる）。

【要確認】秘匿モードをMVPに含めるか、Phase 2に回すか。含める場合の鍵管理（Secure Enclave活用可否）。

### 6.3 ロック／認証
- 起動時に Face ID / Touch ID、フォールバックでアプリ独自パスコード。
- バックグラウンド復帰時の再ロック（タイムアウト設定可）。

### 6.4 PDF書き出し（中核機能）
- カテゴリ単位／全体の書き出し。印刷・共有可能。
- 「秘匿項目を含めない安全版」と「全部入り版」を選択。
- 家族向けに、表紙＋目次＋保管場所一覧を備えた読みやすいレイアウト（PDFKit）。

### 6.5 バックアップ／エクスポート
- **暗号化アーカイブ**（パスフレーズ付き）を書き出し、ファイル/iCloud Driveへユーザー自身が保存。再取り込みで復元。
- 「アプリが将来使えなくなっても、PDFと暗号化アーカイブが手元に残る」を保証する設計。

### 6.6 進捗ダッシュボード
- 記入率の可視化、未記入の重要項目の提案。書き始めのハードルを下げる。

### 6.7 多言語・ローカライズ設計（重要、確度：中）

終活は制度・文化依存が極めて強いため、「翻訳」だけでは不十分。**2層構造**で設計する。

**A層：UI文字列ローカライズ（横展開が容易）**
- 画面ラベル・ボタン・説明文等を **String Catalog（.xcstrings）** で管理。
- ASCの全対応言語に拡張可能。言語追加＝翻訳の追加のみで、コード変更を伴わない構成にする。

**B層：項目テンプレートのロケール出し分け（本質的に重い部分）**
- カテゴリ・項目セット自体を**ロケール別のデータ（テンプレートパック）**として定義（コードに項目をハードコードしない）。
- 構成：**共通ベース項目**（資産・連絡先・メッセージ等、普遍的なもの）＋**ロケール上書き**（JP：本籍/香典/お墓の宗派等、US/UK等：beneficiary、living will、estate executor、probate関連の保管場所等）。
- 各ロケールテンプレートは「項目ラベル＋説明＋並び順＋既定の秘匿フラグ＋免責の差し込み」を持つ。
- テンプレートは JSON/PLIST 等の宣言的データとし、**言語追加＝テンプレートパック＋翻訳の追加**で済む拡張構造にする。

**段階展開**
| 段階 | 言語 | 項目テンプレート | 法的免責 |
|---|---|---|---|
| MVP | ja, en | ja＝作り込み／en＝汎用英語版（西欧一般＋強い免責） | ロケール別に差し込み |
| 拡大（テスト後） | ASC全言語 | まずUI翻訳を全言語へ。項目テンプレは「汎用版→順次現地化」 | 各ロケールで見直し（要レビュー） |

**正直な前提・リスク（確度：中〜低）**
- UI翻訳（A層）は全言語展開が容易。一方、**項目と免責の現地化（B層）は国ごとに制度・法律が異なり、文化的・法的に正確に作るのは重い**。英語圏も US/UK/AU で相続・遺言制度が異なる。
- 対応方針：MVPの en は「**汎用英語版＋『お住まいの国の専門家にご確認ください』の明示**」とし、特定法域への踏み込みを避ける。全言語拡大後も、項目テンプレの現地化は「汎用版を出してから順次精緻化」とする。
- カスタム項目（6.8）が、テンプレートの不足を利用者側で補う安全弁として機能する。

【要確認】各ロケールの項目テンプレと免責文の作成・レビュー体制（特に法域差）。enを「汎用」か「特定国（例：en-US）」のどちらに寄せるか。

### 6.8 カスタム項目（確度：中）

- 利用者が **独自の項目／カテゴリを追加・編集・並べ替え**できる。
- 項目タイプ：テキスト、複数行、日付、選択肢、秘匿（6.2準拠）。
- カスタム項目もPDF／アーカイブ書き出しに反映。
- カスタム項目はロケール非依存（利用者の入力言語のまま保持）。テンプレートが未対応の地域・事情を補完する役割。

【要確認】カスタム項目の上限・並べ替えUX・既定テンプレ項目の非表示（不要項目を隠す）可否。

## 7. 非機能要件

| 区分 | 要件 |
|---|---|
| プライバシー | ネットワーク権限を要求しない。トラッキングなし。データ収集ゼロ。プライバシーラベルは「データ収集なし」。 |
| アクセシビリティ（重要） | Dynamic Type対応、十分なコントラスト、大きいタップ領域、平易な日本語、VoiceOver配慮。 |
| 性能 | 数百項目でも快適。SwiftDataのローカルクエリで完結。 |
| セキュリティ | 端末内暗号化、生体認証、秘匿項目の分離保存。 |
| 信頼性 | データ消失防止（保存の堅牢性、書き出し導線の明示）。 |
| 多言語 | MVPは日本語＋英語。UI翻訳（String Catalog）＋項目テンプレのロケール出し分け（第6.7）。テスト後にASC全言語へ拡大。 |
| アクセシビリティ補足 | 表示言語に応じた文字・レイアウト（英語の語長増にも耐えるUI）。 |

## 8. データモデル（SwiftData スケッチ、確度：中）

```
@Model OwnerProfile        // 本人の基本情報（実質シングルトン）
@Model NoteEntry           // 汎用エントリ：categoryKey, title, structuredFields, freeText,
                           //                status(未/中/完), isSensitive, isCustom, sortOrder, updatedAt
@Model CustomField         // カスタム項目定義：label, fieldType, isSensitive, sortOrder（6.8）
@Model SensitiveBlob       // 秘匿データ：暗号化ペイロード（CryptoKit）, 関連エントリ参照
@Model ContactPerson       // 連絡リスト：name, relation, phone, notifyOrNot
@Model FarewellMessage     // 家族へのメッセージ：recipientLabel, body
@Model DocumentLocation    // 書類のありか：docType, locationHint
@Model AppSettings         // ロック方針・タイムアウト・PDF既定・表示ロケール など
```

- `NoteEntry` を中心に、`categoryKey`（文字列キー）で第6.1の分類を表現。**項目ラベルはデータに持たせず、ロケールテンプレート＋String Catalogから解決**（多言語のため）。
- **項目テンプレートはコード外のデータ**（JSON/PLIST）として保持し、`categoryKey`/`fieldKey`で参照。言語追加時にモデル変更が不要な構造にする。
- `isCustom=true` のエントリと `CustomField` でカスタム項目（6.8）を表現。カスタムのラベルは利用者入力の生文字列（翻訳対象外）。
- 秘匿データのみ `SensitiveBlob` に暗号化分離。
- 保存データは**言語非依存**（キー＋値で保持）。表示時にロケールで解決するため、利用者が表示言語を変えても入力済みデータは保持される。

【要確認】専用エンティティ（BankAccount等）に分けるか、`NoteEntry`の汎用＋structuredFieldsで吸収するか。MVPは後者（軽量）を推奨。

## 9. 画面構成／主要フロー

1. **オンボーディング**（最小）：プライバシー方針（非送信）の明示 → ロック設定 → 完了。
2. **ホーム**：記入率ダッシュボード＋カテゴリ一覧。
3. **カテゴリ画面**：項目リスト（状態バッジ付き）。
4. **項目入力画面**：構造化フィールド＋フリーテキスト。秘匿はトグル＋再認証。
5. **書き出し画面**：PDF／暗号化アーカイブ、秘匿の含有選択。
6. **設定**：ロック方針、タイムアウト、バックアップ、データ削除。

主要フロー：起動→生体認証→ホーム→カテゴリ→入力→（任意）書き出し。

## 11. 法的設計（専門性の限界の明示）

**本アプリは一般的な情報の整理・記録ツールであり、法的・税務的な助言や判断を行わない。** 以下を設計に組み込む。

- 遺言に関する画面では「**エンディングノートは遺言書と異なり法的効力を持たない**」旨を明示。
- 相続・税務・保険等の判断が絡む箇所では結論を出さず、「**専門家（弁護士・司法書士・税理士・FP等）にご相談ください**」と案内（弁護士法72条・税理士法の境界に配慮）。
- 免責文を初回と書き出し時に表示。**ロケールごとに免責を差し込み**、特定法域への踏み込みを避ける（日本＝弁護士法/税理士法、英語版＝「お住まいの国の専門家へ」）。
- ※本章は法的助言ではなく、最終的な可否判断は弁護士等の専門家に確認すること。【要確認】App Store審査（医療・法律関連の表現）での指摘可能性、および各ロケールの免責表現の妥当性。

## 12. 収益モデルと価格

**決定：フリーミアム＋一回買い切りアンロック。価格¥1,500（非消費型IAP）。確度：中**

| 区分 | 内容 |
|---|---|
| 無料 | 全カテゴリの記入・閲覧・ロック・編集・カスタム項目を無制限。 |
| 買い切り ¥1,500（非消費型IAP） | PDF書き出し＋暗号化アーカイブ＋（任意）秘匿モードをアンロック。 |

- **根拠**：紙ノートが1,000〜3,000円（例：コクヨ¥1,713）。無料で書かせて「書き出す瞬間」に価値が立つため転換しやすい。途中課金型と違い、**一回限り・透明**なので「課金不安」訴求と矛盾しない（ヘヤログの設計思想を踏襲）。
- **手数料**：SBPで手取り約85%（日本でもApple IAP前提なら実質15%）。¥100k/月には約79本/月が必要（達成確度は中〜低、ASOが最大変数）。
- **運用**：ローンチ後の転換率を見て¥1,200／¥1,800への調整を検討。

**代替案：完全買い切り（¥1,500前払い）**
- メリット：思想が最もクリーン（最初から全機能・課金なし）。
- デメリット：無料トライアルが無く、混雑カテゴリでDL数が伸びにくい。→ 今回はフリーミアムを採用。

【要確認】秘匿モードを無料に含めるか（安全性 vs 課金価値）。価格はローンチ後に再検証。

## 14. リスクと対応

| リスク | 影響 | 対応 | 確度 |
|---|---|---|---|
| 発見性が低くDLが伸びない | 収益未達 | 「非送信・買い切り・終了に強い」を軸にしたASO、終活関連の文脈訴求 | 中 |
| 既存オフライン競合との同質化 | 差別化弱化 | UI品質・項目設計・PDF品質で明確に上回る | 中 |
| 機微情報保存に伴う安全責任 | 信頼毀損 | 既定は「ありかモード」、秘匿は暗号化＋警告、書き出し既定で伏せる | 中 |
| クラウド不在による不便（バックアップ等） | 離脱 | PDF＋暗号化アーカイブ＋iCloud Drive手動保存で補完 | 中 |
| 審査（医療/法律表現） | リジェクト | 結論回避・免責・専門家案内を徹底 | 低〜中 |
| サービス継続性（個人開発） | ユーザー不安だが、むしろ訴求点 | 「事業者依存しない＝終了しても端末とPDFに残る」を価値として明示 | 高 |
